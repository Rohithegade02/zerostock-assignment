# Database-Focused Assignment: Inventory + Suppliers

This project implements the B: Database-Focused Assignment using Node.js, Express, and MongoDB.

## Running Locally

1. Make sure you have MongoDB running locally at `mongodb://localhost:27017` or supply a `MONGO_URI` in an `.env` file.
2. Install dependencies: `npm install`
3. Start the dev server: `npm run dev`

Server runs on `http://localhost:8001`.

## Endpoints

1. **POST /supplier**
   * Creates a new supplier.
   * Body: `{ "name": "Supplier A", "city": "Mumbai" }`

2. **POST /inventory**
   * Creates new inventory for a supplier.
   * Rules enforced: Valid supplier `supplier_id`, `quantity >= 0`, `price > 0`.
   * Body: `{ "supplier_id": "<mongo_id>", "product_name": "Keyboard", "quantity": 10, "price": 100 }`

3. **GET /inventory**
   * Aggregation query that returns all inventory grouped by supplier.
   * **Required logic:** Sorted by the *total inventory value* (quantity × price).

## Database Architecture & Design Choices

### Database Schema Explanation

We have two main collections in MongoDB:

1. **Suppliers**:
   * `name`: String
   * `city`: String
   * `_id`: Automatically generated by MongoDB. Serves as the primary key.

2. **Inventory**:
   * `supplier_id`: ObjectId (References `Suppliers._id`)
   * `product_name`: String
   * `quantity`: Number (Min: 0)
   * `price`: Number (Min: 0, Exclusive)

**Relationship**: A **One-to-Many** relationship is implemented via referencing. Each `Inventory` document holds a `supplier_id` reference to the `Suppliers` collection.

### Why SQL vs NoSQL (MongoDB)

For an Inventory and Supplier relationship, both SQL and NoSQL are viable. In a strictly relational dataset, SQL (like PostgreSQL with explicit foreign keys) is often a strong default to strictly enforce integrity lock-ins.

However, I chose **MongoDB (NoSQL)** for this assignment because:
1. **Schema Flexibility**: If products begin having highly variable metadata (e.g., shoe size vs. electronic wattage), NoSQL naturally allows unstructured arbitrary fields without creating complex sparse tables or JSON blobs in SQL.
2. **Powerful Aggregation Framework**: Grouping by supplier and computing arithmetic properties (`quantity * price`) directly translates to an elegant MongoDB `$group` and `$sort` pipeline.
3. **Mongoose Middleware**: Enforcing validation (quantity >= 0, price > 0, verifying supplier exists) is extremely straightforward at the ODM layer. 
4. **Fast Prototyping**: MongoDB works cohesively with Node.js applications, allowing rapid feature expansion.

### Indexing & Optimization Suggestion

To optimize performance (especially as the `Inventory` collection grows), we should add an **index on the `supplier_id` field in the Inventory collection.**

```javascript
// Example in Mongoose:
inventorySchema.index({ supplier_id: 1 });
```

**Why?** 
* The `GET /inventory` groups records by `supplier_id`. Aggregation `$group` operations without an index run a full collection scan which is an `O(N)` operation.
* An index on `supplier_id` will dramatically speed up `lookup` and grouping operations.
* If products are frequently queried by name, a compound index `{ supplier_id: 1, product_name: 1 }` or a text index on `product_name` would be the next optimal step.
